## 各ファイルが何のためにあるか

### main.py — 投稿のアイデアをAIに出してもらう

Gemini APIを呼び出して、X（Twitter）の投稿候補を自動生成します。結果は `posts.csv` に保存されます。スクリプトを1回走らせると、今日投稿できそうな文章が複数出てくる。それを見て、一番しっくりくるのを選んでXに手動投稿する、という使い方です。

「毎日何を投稿しようか」を0から考えるのは消耗します。AIに選択肢を出してもらって、自分は選ぶだけにしたかった。そのためのスクリプトです。

---

### daily_run.py — 1日の流れを1コマンドで

```
python daily_run.py
```

これを実行するだけで、
- 最近の投稿履歴を表示
- X投稿の候補を生成
- サイトを最新状態に更新
- 今日の収益アクションを表示
- note記事の下書きを確認

…という一連の作業が順番に走ります。

「今日何をやるか」を毎朝考えなくて済むように、全部このコマンドに集約しました。複雑な仕組みに見えますが、やっていることは「いくつかのスクリプトを順番に呼ぶ」だけです。

---

### build_site.py — CSVからサイトを組み立てる

`site/products.csv` に商品情報を書いておくと、このスクリプトがHTML形式のサイトを組み立てて `site/index.html` と `docs/index.html` を更新します。

商品を追加したいとき、HTMLを直接触らなくていい。CSVに1行書いて、スクリプトを実行するだけ。こういう「データとデザインを分けておく」設計が、後で何かを変えるときにラクになります。

---

### affiliate_tools.py — リンクの確認と整理

アフィリエイトリンクを一覧表示したり、アフィリエイトタグ（`tag=xxxxx`）が全リンクに正しく付いているかを確認したり、アフィリエイト開示文がページに設置されているかをチェックするツールです。

Amazonアソシエイトの規約では、アフィリエイトリンクがあるページには必ず開示文が必要です。「うっかり抜けてた」を防ぐために作りました。

---

### site/ と docs/ — なぜフォルダが2つあるか

`site/` がソースフォルダ（自分が直接編集するもの）、`docs/` がGitHub Pagesに公開されるフォルダです。

GitHubの設定で「docsフォルダをWebに公開する」に設定しておくと、`docs/` の中身がそのまま `https://ユーザー名.github.io/リポジトリ名/` として公開されます。サーバーもドメイン代も要らない。無料です。

`site/` で内容を作って、`docs/` にコピーする（または `build_site.py` が自動でやる）という流れです。

---

### site/products.csv — 商品データを1か所にまとめる

商品名・説明・AmazonのURL・カテゴリー・アフィリエイトタグ…といった情報をCSV形式で管理しています。

HTMLの中に商品情報を直接書いてしまうと、後から修正するときにどこを探せばいいか分からなくなります。「データは1か所に」という原則を守っておくと、後がラクです。

---

### note_drafts/ — 日本語noteの下書き置き場

日本語で書いたMarkdown形式の記事が、ここに貯まっていきます。ファイル名は `YYYY-MM-DD-タイトルのスラグ.md` という形式。今読んでいるこの記事も、このフォルダに保存されています。

無料記事・有料記事はファイルの先頭に `type: free` または `type: paid` と書いておくことで管理しています。投稿は手動です。スクリプトは絶対に自動投稿しません。

---

### posts.csv と top_posts.csv — X投稿の候補管理

`main.py` が生成したX投稿の全候補が `posts.csv` に入ります。そこからスコアや関連性で絞り込んだ上位数件が `top_posts.csv` に入ります。毎朝 `daily_run.py` を実行すると、ターミナルに今日の候補が表示されます。

---

### README.md — 自分向けの取扱説明書

「このプロジェクトは何をするものか」「どのスクリプトを何のために使うか」を書いておくファイルです。

GitHubに公開することを意識して書いてますが、正直、一番のターゲットは「3ヶ月後の自分」です。時間が経つと「このスクリプト、なんで作ったんやっけ」となるので。

---

### .env と .env.example — APIキーをどう管理するか

**大事な話をします。絶対に `.env` をGitHubに上げてはいけません。**

Gemini APIを使うためには、Googleから発行された「APIキー」が必要です。このキーは、言ってみれば「あなた専用のパスワード」です。GitHubのパブリックなリポジトリにアップしてしまうと、誰でも見られる状態になって、不正利用されるリスクがあります。

対策はシンプルです。

**`.env` ファイルにAPIキーを書く。そして `.gitignore` に `.env` を追加してGitに含めないようにする。**

```
# .env（このファイルはGitに含めない）
GEMINI_API_KEY=AIzaSy...実際のキー...
```

```
# .env.example（こちらはGitに含める）
GEMINI_API_KEY=your_api_key_here
```

`.env.example` は「こういうキーが必要ですよ」というサンプルを、安全に共有するためのファイルです。実際のキーは入れません。

## Gemini APIをどう使っているか

Google の Gemini API は、無料枠の範囲内でも使えます。

このプロジェクトでは主に2つの用途で使っています。

**1. X投稿の候補生成**
旅行に関するテーマを渡すと、投稿として使えそうな文章を複数生成してもらいます。全部採用するわけではなく、「このトーンは合ってる」「この内容は今日に合う」と感じたものを選ぶ。最後の判断は自分がします。

**2. 記事・note下書きの生成**
テーマ・方向性・シリーズのトーンを指示として渡すと、Markdown形式の下書きを生成してもらいます。そのまま投稿するのではなく、必ず自分で読んで書き直してから使います。

Gemini に限らず、AI生成の文章をそのまま使うのは気をつけた方がいいです。読めば分かりますが、「自分の体験が入っていない」文章は、読む人にも伝わります。

## GitHub Pagesでどう公開しているか

「サーバー代が要らない」のがGitHub Pagesの最大のメリットです。

手順としては：

1. GitHubにリポジトリを作る
2. リポジトリの設定で「Source: docs フォルダ」を選ぶ
3. `docs/` フォルダにHTMLファイルを置く
4. それだけで `https://ユーザー名.github.io/リポジトリ名/` に公開される

独自ドメインを持っていなくても、自分のサイトが公開できます。

ただし、「Gitに変更をpushするたびに公開内容が更新される」という仕組みなので、ファイルを直接クラウドに保存するのではなく、**ローカルで編集 → gitにコミット → pushで公開**という流れになります。最初はここが一番とっつきにくいかもしれません。でも慣れると、コマンド3行で更新が反映されるのは快適です。

## note下書きが収益ワークフローにどう入るか

このプロジェクトの収益の柱は今2本です。

**1. Amazonアフィリエイト**（英語サイト経由）
旅行準備の記事やチェックリストツールから、商品ページへのアフィリエイトリンクをクリックしてもらい、購入が発生したときに収益が入る仕組み。

**2. note有料記事**（日本語）
このシリーズの有料記事を投稿して、読者に購入してもらう。¥300〜¥500の小さな単価ですが、シリーズが続けばファンになってもらえる可能性がある。

`note_drafts/` に保存された有料記事ファイルは、`data/monetization_log.csv` に記録されて、`daily_run.py` の「今日のマネタイズルート」セクションに表示されます。「今日投稿できる有料記事がある」「今日は動画をアップできる」というのが、毎朝コマンドを実行したときに見えるようになっています。

完璧に自動化できているわけではありません。でも「今日何をするか迷わない」仕組みは、続けるうえで思った以上に大事です。

## 僕が最初に作ってよかったファイルたち

たくさんファイルがあると、「どれが本当に必要だったのか」が分かりにくいです。正直に言うと、最初から必要だったのは以下の5つです。

**1. `index.html`（サイトの入り口）**
これがなければ何も始まらない。見た目が悪くてもいいので、まず公開する。

**2. `products.csv`（商品データ）**
アフィリエイトリンクをここに集約しておくと、後から管理が楽になる。最初から「データとHTMLを分ける」設計にしておいてよかった。

**3. `.env` と `.gitignore`**
APIキーの管理をミスると取り返しがつかない。最初から正しく設定しておく。

**4. `daily_run.py`（1日の作業ドライバー）**
毎日「今日何をしよう」と考えることがなくなった。「実行して、選んで、投稿する」の繰り返しに集中できる。

**5. `README.md`**
「このプロジェクトは何をするものか」を書き留めておかないと、すぐ迷子になる。3ヶ月後の自分のために書く。

## 失敗しやすいポイント

**1. 仕組みを作ることが目的になる**

スクリプトを書いたり、フォルダを整理したりすること自体が楽しくなって、肝心の「コンテンツを作る・投稿する」がおろそかになる罠です。仕組みは手段であって、目的じゃない。

**2. コミットを怠る**

Gitの変更を「後でまとめてコミット」にすると、どこで何を変えたか分からなくなります。小さい変更でもこまめにコミットする習慣をつける。

**3. APIキーを間違えてコミットする**

やってしまうと、GitHubに取り消せない形で残ります。`.gitignore` に `.env` が入っているか、毎回確認する。心配なら `git status` でコミット前に確認する。

**4. サイトと docs/ の同期を忘れる**

`site/` を編集しても `docs/` にコピーしないと、公開サイトは古いままです。`build_site.py` を実行してから `docs/` をpushするのをセットで覚える。

**5. 「まだ完成してない」でいつまでも公開しない**

完成したときに公開しよう、は永遠に来ないです。走りながら直すのが唯一の正解。
